§ 第一部分：开篇与背景

第1章：Multica 是什么？

目标：建立对 Monica 系统的整体认知，理解它解决什么问题

1.1 AI-Native 时代的软件开发

传统软件开发模式正面临新的挑战。随着 AI Agent 技术的发展，开发者不再是唯一的代码创造者。AI 可以协助编写代码、调试程序、甚至自主完成完整的开发任务。这带来了新的工作方式，但也带来了新的管理挑战。

  • 传统模式的局限性：单线程工作、上下文切换成本高、知识难以复用
  • AI Agent 的改变：并行执行、24/7 工作、技能累积
  • 管理平台的必要性：协调多个 Agent、追踪进度、处理阻塞

1.2 Multica 的核心理念

Multica 的核心理念是"AI as Teammate" —— 将 AI Agent 视为团队的一员，而不是工具。

  • Agent 作为队友：可以分配任务、创建 issue、发表评论、报告进度
  • Squad 机制：将多个 Agent 组织成团队，由 Leader Agent 进行任务分配
  • 技能复用：每个解决方案都成为团队可复用的技能

1.3 支持的 Agent 类型

Agent 类型 类型 支持状态
Claude Code 官方 CLI 支持
Codex OpenAI CLI 支持
GitHub Copilot CLI GitHub CLI 支持
OpenClaw / OpenCode 开源 CLI 支持
Gemini / Pi / Kimi 第三方 CLI 支持
Kiro CLI 第三方 CLI 支持

§ 第二部分：技术栈详解

第3章：后端技术栈 - Go + Chi + sqlc

目标：理解 Monica 后端选择 Go 的原因，以及关键框架的使用

3.1 为什么选择 Go

  • 编译型性能：编译成机器码，启动快、执行快，适合高并发场景
  • 并发模型：Goroutine + Channel 提供轻量级并发，适合处理大量 Agent 任务
  • 标准库丰富：内置 HTTP、JSON、加密等功能，减少第三方依赖
  • 错误处理：显式错误返回，避免隐藏的运行时异常

3.2 Chi 路由框架

Chi 是一个轻量级、高性能的 HTTP 路由框架，设计理念是"少即是多"。

// server/cmd/server/router.go - 路由配置示例
func NewRouter(pool *pgxpool.Pool, hub *realtime.Hub, ...) chi.Router {
    r := chi.NewRouter()
    
    // 全局中间件
    r.Use(chimw.RequestID)
    r.Use(middleware.ClientMetadata)
    
    // API 路由组
    r.Route("/api", func(r chi.Router) {
        r.Use(middleware.RequireAuth)
        r.Use(middleware.RequireWorkspace)
        
        r.Get("/issues", h.ListIssues)
        r.Post("/issues", h.CreateIssue)
        // ...
    })
    
    return r
}

3.3 sqlc - 类型安全的 SQL

sqlc 是一个从 SQL 生成类型化 Go 代码的工具，它直接将 SQL 查询转换为类型安全的函数。

-- server/pkg/db/queries/issue.sql - sqlc 查询定义
-- name: GetIssue :one
SELECT id, title, description, status, priority, 
       assignee_type, assignee_id, workspace_id,
       created_at, updated_at
FROM issues 
WHERE id = $1;

-- name: CreateIssue :one
INSERT INTO issues (
    id, workspace_id, title, description, 
    status, priority, assignee_type, assignee_id
) VALUES (
    $1, $2, $3, $4, $5, $6, $7, $8
) RETURNING *;

sqlc 会自动生成对应的 Go 代码：

// sqlc 生成的代码示例（简化）
type GetIssueParams struct {
    ID uuid.UUID
}

type Issue struct {
    ID          uuid.UUID
    Title       string
    Description sql.NullString
    Status      string
    // ...
}

func (q *Queries) GetIssue(ctx context.Context, params GetIssueParams) (Issue, error)
func (q *Queries) CreateIssue(ctx context.Context, params CreateIssueParams) (Issue, error)
📊 sqlc vs ORM 对比
维度 sqlc GORM
类型安全 编译时检查 运行时检查
性能 原生 SQL，无额外开销 有 ORM 层开销
灵活性 完全控制 SQL 受限于 ORM API

6.1 WebSocket Hub 架构

Monica 使用 gorilla/websocket 实现 WebSocket 通信，支持两种模式：

  • 单节点模式：内存 Hub，适合开发和小型部署
  • 多节点模式：Redis 中继，支持水平扩展
// server/internal/realtime/hub.go - Hub 核心结构
type Hub struct {
    clients    map[*Client]map[string]bool  // client -> subscribed rooms
    rooms      map[string]map[*Client]bool  // room -> subscribed clients
    register   chan *Client
    unregister chan *Client
    broadcast  chan *Message
}

func (h *Hub) Run() {
    for {
        select {
        case client := <-h.register:
            // 处理客户端注册
            h.registerClient(client)
        case client := <-h.unregister:
            // 处理客户端注销
            h.unregisterClient(client)
        case msg := <-h.broadcast:
            // 广播消息到房间
            h.broadcastToRoom(msg)
        }
    }
}

11.3 API 响应兼容性

Monica 使用 Zod Schema 进行 API 响应验证，确保前端不会因为后端变更而崩溃：

// packages/core/api/schema.ts
export const IssueSchema = z.object({
    id: z.string().uuid(),
    title: z.string(),
    status: z.enum(['backlog', 'todo', 'in_progress', 'review', 'done', 'archived']),
    priority: z.number().int().optional(),
    // 新字段必须可选，确保向后兼容
    newField: z.string().optional(),
});

export const parseWithFallback = (
    data: unknown,
    schema: ZodType,
    fallback: T
): T => {
    const result = schema.safeParse(data);
    if (!result.success) {
        console.warn('API response parsing failed:', result.error);
        return fallback;
    }
    return result.data;
};

§ 第15章：常见问题与解决方案

15.1 开发环境问题

问题 解决方案
Docker 容器启动失败 检查 Docker Desktop 是否运行，端口 5432 是否被占用
端口冲突 使用 `make worktree-env` 创建独立环境
数据库迁移失败 运行 `make db-reset` 重置数据库

15.2 调试技巧

  • 后端日志分析：查看 `slog` 输出，关注请求 ID
  • 前端调试：使用 React Query DevTools 查看缓存状态
  • WebSocket 追踪：在浏览器开发者工具的 Network 面板查看 ws 消息
